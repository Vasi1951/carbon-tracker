# ADR 009: Production Architecture and Disaster Recovery (DR) Strategy

## Status

Accepted

## Context

Deploying the Carbon Tracker to production necessitates a robust, secure, and highly available architecture capable of handling fluctuating loads while protecting sensitive user environmental data. Furthermore, we must establish a clear Disaster Recovery strategy to define our Recovery Point Objective (RPO) and Recovery Time Objective (RTO).

### Requirements

- **High Availability**: 99.99% uptime SLA.
- **RPO (Recovery Point Objective)**: 1 Hour. We cannot afford to lose more than 1 hour of user activity logs.
- **RTO (Recovery Time Objective)**: 30 Minutes. The system must be fully operational within 30 minutes of a critical failure.
- **Security**: Defense-in-depth, least privilege IAM, and strict supply chain security (Binary Authorization).
- **Compliance**: Data retention mandates hot storage for 1 year and cold storage for 7 years.

## Decision

We have architected a Google Cloud Platform (GCP) native infrastructure using Terraform, governed by GitHub Actions.

### 1. Infrastructure Architecture

- **Compute**: Cloud Run (API and Web). Chosen for scale-to-zero capabilities (cost optimization) and massive burst scaling (1 to 100 instances). `cpu_idle = false` ensures CPUs are always allocated to eliminate cold-start latency for the API.
- **Database**: Cloud SQL for PostgreSQL 16 (Regional HA). Deployed in a private subnet with automated backups and Point-In-Time Recovery (PITR) enabled.
- **Caching**: Cloud Memorystore for Redis 7 (Standard HA tier) to cache Gemini insights and high-frequency queries.
- **Network & Edge**: Global External HTTPS Load Balancer with Cloud CDN for static Web assets, fronted by Cloud Armor (WAF) to mitigate DDoS and common web exploits.

### 2. Deployment Pipeline (CI/CD)

- **Multi-Stage Distroless Builds**: Application images are built using `gcr.io/distroless/nodejs20-debian12`, stripping away shells and package managers to minimize the attack surface.
- **Vulnerability Scanning**: Trivy scans images pre-deployment. Any `CRITICAL` or `HIGH` vulnerabilities automatically halt the pipeline.
- **Canary Deployments**: We route 10% of traffic to new deployments for 30 minutes. Cloud Monitoring alerts (5xx > 1%) trigger automated rollbacks.
- **Binary Authorization**: Only images pushed from our trusted GitHub Actions pipeline (signed via Artifact Registry) are admitted by Cloud Run.

### 3. Disaster Recovery (DR) Strategy

- **Database Backups (RPO: 1 Hour)**: Cloud SQL automated backups occur daily. PITR is enabled, utilizing Write-Ahead Logs (WAL) to guarantee an RPO of mere minutes, easily satisfying the 1-hour requirement.
- **Automated Failover (RTO: 30 Minutes)**: Cloud SQL Regional HA automatically fails over to the standby instance in < 60 seconds if the primary zone degrades. Cloud Run and Load Balancing are inherently multi-zonal/global.
- **Data Retention**: Database snapshots older than 1 year will be exported to Cloud Storage Archive class for the required 7-year cold retention period.
- **Backup Testing**: Monthly automated GitHub Action to restore the production DB snapshot into an isolated staging environment to verify backup integrity.

## Consequences

### Positive

- Exceptional scalability without operational overhead (No Kubernetes clusters to manage).
- Ironclad supply chain security via Distroless and Binary Auth.
- Guaranteed RPO/RTO compliance.

### Negative

- Vendor lock-in to GCP specific managed services (Cloud Run, Memorystore).
- Increased complexity in the CI/CD pipeline due to Canary bake times and Trivy scanning.
