# ADR-001: Clean Architecture with Bounded Contexts

## Status

Approved

## Context

We need to design a scalable carbon footprint tracking application (`carbon-tracker`) that remains maintainable as business logic grows and infrastructure details (such as database types, API layers, cache servers) change.

## Decision

We adopt **Clean Architecture** principles structured as a monorepo containing distinct packages and applications with strict dependency directions:

```
[Apps (web, mobile, api)]
          │
          ▼
    [Infrastructure]
          │
          ▼
    [Application]
          │
          ▼
       [Domain]
```

- **Domain**: Contains pure business logic and entities. It has zero external dependencies (no frameworks, no databases).
- **Application**: Contains use cases and ports (interfaces representing external contracts).
- **Infrastructure**: Implementations of database clients (Prisma, PostgreSQL), caching (Redis), mail services, etc. High-level policies from `Application` must not directly depend on this. Instead, infrastructure adapters implement ports defined in `Application` (Dependency Inversion).
- **Shared-types**: Houses common type signatures that span multiple apps and packages.

### Import Constraints

- `/domain` has no dependencies and must not import from any other project packages.
- `/application` can only import from `/domain` and `/shared-types`.
- `/infrastructure` must not import from `/domain` directly (enforcing Dependency Inversion via `/application` ports). It is restricted via custom ESLint boundaries.

## Consequences

- **Pros**: Easy testing of core domain logic without mocking databases. Databases/Frameworks can be replaced with minimal impact on business use cases.
- **Cons**: Requires more boilerplate files (ports, interfaces, adapters) and strict typing mapping between layers.
