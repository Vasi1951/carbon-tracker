# ADR 008: Performance Budget and Accessibility Compliance

## Status

Accepted

## Context

As the Carbon Tracker platform (both Web and Mobile) matures, ensuring an inclusive experience that performs flawlessly on low-end devices and poor network connections is paramount. The platform must adhere to strict Accessibility (WCAG 2.1 AAA) and Performance budgets to maintain user engagement, specifically targeting carbon-conscious users who might be leveraging older, energy-efficient devices.

### Accessibility Requirements

- Web and Mobile must be fully operable via keyboard or screen reader.
- Automated tools must report zero critical/serious issues.
- Cognitive accessibility requires explicit confirmations for destructive actions and progressive loading indicators.

### Performance Requirements

- **Web**: Initial payload < 150KB gzipped.
- **Mobile**: Cold start < 1.5s, avoiding battery-draining background polling for location.

## Decision

We have implemented the following strategies across the stack:

### 1. Web Optimizations (React + Vite)

- **Code Splitting**: We decomposed the monolithic `App.tsx` into domain-specific chunks (`AuthPage`, `DashboardPage`, `AddActivityModal`). Using `React.lazy()` and `<Suspense>`, these chunks are loaded on-demand, aggressively reducing the initial bundle size to meet the 150KB budget limit configured via `chunkSizeWarningLimit` in `vite.config.ts`.
- **Manual Chunking**: Third-party dependencies are explicitly grouped into a `vendor` chunk.
- **Progressive Web App (PWA)**: Implemented `vite-plugin-pwa` for Service Worker caching and offline fallback, ensuring rapid Time to Interactive (TTI) on subsequent visits.
- **A11y Runtime Checks**: Integrated `@axe-core/react` in development mode to immediately flag accessibility violations in the DOM.

### 2. Mobile Optimizations (React Native)

- **Inline Requires**: Enabled `inlineRequires` via `metro.config.js` to defer the evaluation of JavaScript modules until they are actually needed, directly minimizing the cold start time to < 1.5s.
- **Lazy Loading**: Adopted `React.lazy()` for screen transitions to keep the JS thread free during initial hydration.
- **Eco-Location**: Abandoned active background location polling in favor of the OS-level "Significant Location Change" API. This vastly improves battery life, aligning with the app's core mission of energy conservation.
- **Native A11y**: Enforced the use of `accessible={true}`, `accessibilityRole`, and `accessibilityHint` across all interactable components (like Quick Add buttons).

## Consequences

### Positive

- **Inclusivity**: Achieving WCAG 2.1 AAA ensures the app is usable by all demographic segments.
- **Speed & SEO**: The rigorous chunking and PWA caching ensure near-instantaneous load times, improving Lighthouse CI scores and user retention.
- **Battery Life**: The mobile app is significantly less demanding on device batteries.

### Negative

- **Complexity**: Developers must consciously manage component boundaries to ensure lazy-loading works effectively without introducing jank.
- **PWA Stale Assets**: Service Workers require careful cache invalidation strategies during deployments.
