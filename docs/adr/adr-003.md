# ADR-003: Use Case Boundaries and Port Definitions

## Status

Approved

## Context

In our Hexagonal (Ports and Adapters) architecture, the application core holds the business use cases and defines the interfaces (ports) required for interacting with the outside world (databases, message brokers, caching systems, etc.). We need a formal contract defining these use cases and port boundaries to:

1. Ensure the core application layer remains framework-agnostic (no Express, React, or database ORM dependencies).
2. Establish strict compile-time boundaries for third-party service integration.
3. Enforce the `Result<T, E>` pattern to prevent throwing business exceptions.
4. Support 100% test coverage using mock adapters.

## Decision

We define three core application use cases and five port interfaces:

### Use Cases

1. **`RecordActivityUseCase`**
   - **Input**: `{ userId, category, amount, unit, date, description }`
   - **Flow**: Validates inputs using domain entities, fetches the emission factor, calculates carbon footprints (`kgCO2e`), saves the activity via repository, and publishes an `ActivityRecorded` event.
   - **Output**: `{ activityId, kgCO2e, message }` wrapped in `Result`.

2. **`GetUserDashboardUseCase`**
   - **Input**: `{ userId, period }` (day, week, month, year)
   - **Flow**: Checks cache for pre-aggregated dashboard. On cache miss, fetches activities within the timeframe, queries emission factors, aggregates emissions per category, calculates percentages, compares totals against the current goal to compute trends (up/down/stable), and caches the output.
   - **Output**: `{ totalKgCO2e, breakdown: [{ category, amount, percentage }], trend }` wrapped in `Result`.

3. **`SetCarbonGoalUseCase`**
   - **Input**: `{ userId, targetKgCO2e, timeframe }`
   - **Flow**: Computes 30-day average emissions. Validates that the target is realistic (not < 10% of current average). If valid, saves the goal and determines the goal target date based on timeframe.
   - **Output**: `{ goalId, target, projectedDate }` wrapped in `Result`.

### Ports (Interfaces)

- **`IActivityRepository`**: Interacts with the activity store (`save`, `findByUserAndPeriod`, `delete`).
- **`IEmissionFactorRepository`**: Fetches and caches emission factors (`findByCategoryAndRegion`, `cacheFactors`).
- **`IUserGoalRepository`**: Manages goal storage (`saveGoal`, `getCurrentGoal`).
- **`ICacheService`**: Provides transient key-value caching (`get`, `set`, `del`).
- **`IEventBus`**: Publishes asynchronous integration events (`ActivityRecorded`, `GoalAchieved`, `MilestoneReached`).

## Consequences

- **Decoupled Business Logic**: Third-party framework upgrades (e.g. Prisma or Redis client versions) will not require modifications to application use cases.
- **Robust Exception Handling**: Use cases return typed `Result` objects. The API layer (Express controller) does not have to deal with nested try-catch blocks for validation/business rule exceptions.
- **Test-Friendly Architecture**: Tests are fast and run purely in-memory using stub repositories, cache, and event bus implementations.
