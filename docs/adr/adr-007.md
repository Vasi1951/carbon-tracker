# ADR 007: Quality Gate Strategy and Tooling

## Status

Accepted

## Context

As the Carbon Tracker platform scales, maintaining high code quality across multiple packages (domain, application, infrastructure, web API, mobile, etc.) becomes challenging. Different layers of the application have different requirements, but overall, we require a zero-tolerance approach to code quality regressions. We need an automated strategy that catches defects as early as possible without overly hindering developer velocity.

We needed to decide on the tooling and rules for:

1. Static analysis and linting.
2. Unit and Integration test coverage.
3. Mutation testing to prove test suite efficacy.
4. Pre-commit hooks to block bad code from entering the repository.

## Decision

We have implemented a comprehensive Quality Gate Strategy relying on the following principles and tools:

### 1. Strict Static Analysis (ESLint & TypeScript)

- We enforce strict ESLint rules globally, including `@typescript-eslint/strict-type-checked`.
- **Zero-Tolerance Rules**: Disallow `any` type (`@typescript-eslint/no-explicit-any`), prevent floating promises, and forbid confusing void expressions.
- **Architectural Boundaries**: Use `eslint-plugin-import`'s `no-restricted-paths` to enforce Dependency Inversion. For instance, `/domain` cannot import from `/infrastructure`.
- **Pragmatic Overrides**: While `max-lines-per-function` and `complexity` rules are enforced globally to encourage small, testable functions, these are explicitly disabled for React UI components (`**/*.tsx`) and test files (`**/*.test.ts`). This recognizes that declarative UI and setup-heavy tests naturally result in longer functions.

### 2. Comprehensive Test Automation & Coverage Requirements

- **Vitest**: Used as the primary unit testing framework.
- **Coverage Thresholds**:
  - `domain`: 100% coverage required.
  - `application`: 90% coverage required.
  - `infrastructure`: 85% coverage required.
- **Integration Tests**: `testcontainers` is used for spinning up real PostgreSQL and Redis instances for API integration tests. This avoids brittle mocks for infrastructure boundaries.

### 3. Mutation Testing

- **Stryker**: Adopted to perform mutation testing. Stryker seeds faults into the application code and checks if the test suite catches them. This ensures that 100% coverage isn't just executing code without meaningful assertions.
- We require a minimum mutation score of 80% to merge any PR.

### 4. Commit and Merge Blocking (Husky & Lint-Staged)

- **Husky**: Used for pre-commit hooks to run `lint-staged`.
- **lint-staged**: Runs ESLint and Prettier exclusively on staged files before a commit is allowed. Any linting error (including the use of `any` without an override) will block the commit.
- **CI Pipeline**: The CI pipeline is configured to fail the build if `npm run lint` or `npm run test` fail, blocking PR merges.

## Consequences

### Positive

- High confidence in the correctness and maintainability of the codebase.
- Clear architectural boundaries are automatically enforced, preventing "big ball of mud" degradation.
- Real dependencies used in integration tests reduce production surprises.

### Negative

- Slower initial development speed, as developers must satisfy strict type checks and high coverage thresholds.
- Mutation testing increases the overall time taken by the CI pipeline, requiring careful caching strategies.
- Pragmatic overrides for ESLint (like disabling complexity on `.tsx` files) might occasionally allow overly complex UI components to merge if not caught in code review.
